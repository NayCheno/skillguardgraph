# Evidence Graph Specification

This document specifies the typed evidence graph used by SkillGuardGraph for cross-layer security analysis. It covers the node and edge schema, the constraint library (C1--C7), the evidence path serialization format for audit logs, and the graceful degradation strategy when certain evidence sources are unavailable.

---

## 1. Schema

### 1.1 Node Types

The evidence graph contains nine node types. Each node has a unique identifier (`id`), a type tag (`node_type`), and a properties dictionary. The following table defines each type, its purpose, and its required and optional properties.

| Node Type | Identifier Convention | Required Properties | Optional Properties | Description |
|---|---|---|---|---|
| **Skill** | `skill:<name>` | `name`, `platform_type` | `publisher`, `description`, `install_date`, `trust_tier` | Represents a skill artifact (MCP server, connector, action, tool, plugin, extension). The primary entity in the graph. |
| **Version** | `version:<name>@<semver>` | `semver`, `release_date` | `changelog`, `signature`, `hash`, `fingerprint` | A specific version of a skill. Tracks temporal evolution and enables drift detection across updates. |
| **Metadata** | `metadata:<name>:<field>` | `field` (e.g., `description`, `schema`, `annotations`, `scope`), `value` | `source` (manifest, marketplace, publisher), `confidence` | A declared metadata assertion from the skill's manifest, description, schema annotations, or marketplace listing. |
| **CodeSlice** | `code:<name>:<component>` | `component` (e.g., `handler`, `dependency`, `entrypoint`), `language` | `sinks`, `sources`, `taint_summary`, `capabilities`, `obfuscation_score` | A static analysis artifact representing a code component, inferred capabilities, and conservative source--sink summaries. The current prototype combines regex capability extraction with a Python AST pass for common import aliases and variable-to-sink flows; it is not a full interprocedural taint analyzer. |
| **RuntimeEvent** | `event:<call_id>` | `event_type` (e.g., `tool_call`, `response`, `network_request`, `file_write`, `memory_write`), `timestamp` | `arguments`, `return_value`, `duration_ms`, `session_id`, `caller`, `callee` | A single observed event during skill execution: a tool call, a network request, a file operation, or a memory write. |
| **DataObject** | `data:<hash>` or `data:<name>` | `data_type` (e.g., `document`, `token`, `file`, `email`, `webpage`, `structured`) | `content_preview`, `size_bytes`, `classification`, `source_system` | A piece of data that flows through the system: a document, credential, file, email, or webpage content. |
| **TrustLabel** | `trust:<level>` | `level` (e.g., `internal`, `approved_third_party`, `unknown`, `untrusted`) | `basis` (e.g., `signature`, `reputation`, `policy`, `manual`) | A trust classification assigned to a skill, data source, or content origin. Propagates along `flows_to` edges. |
| **SensitivityLabel** | `sensitivity:<level>` | `level` (e.g., `public`, `internal`, `confidential`, `restricted`, `secret`) | `classification_system`, `classifier_confidence` | A data sensitivity classification applied to a DataObject. Used to determine whether exfiltration constraints are triggered. |
| **PolicyDecision** | `decision:<id>` | `decision` (e.g., `allow`, `degrade`, `hitl`, `sandbox_only`, `deny`, `quarantine`, `rollback`), `constraint`, `timestamp` | `rationale`, `evidence_path`, `override_by`, `override_reason` | A policy enforcement outcome generated by evaluating constraints against the graph. Links the finding to its supporting evidence. |

### 1.2 Edge Types

Edges connect nodes and encode relationships. Each edge has a source node, target node, type tag (`edge_type`), and optional properties.

| Edge Type | Source Node(s) | Target Node(s) | Required Properties | Description |
|---|---|---|---|---|
| **declares** | Skill, Version | Metadata | `field` | The skill or version declares a metadata assertion (capability, description, schema). |
| **implements** | Version, CodeSlice | Metadata, Skill | `confidence` | The code implements or supports the declared or inferred capability. |
| **observes** | RuntimeEvent | Metadata, CodeSlice, Skill | `confidence`, `source` (sandbox or runtime) | Sandbox probing or runtime monitoring observed a behavior that corresponds to a declared or inferred capability. |
| **flows_to** | DataObject, RuntimeEvent, TrustLabel | DataObject, RuntimeEvent, LLMContext, PersistentStore | `mechanism` (e.g., `tool_return`, `context_injection`, `file_read`, `memory_recall`) | Data, instructions, or trust labels flow from one entity to another. This is the primary edge for tracking information propagation. |
| **calls** | Skill, RuntimeEvent | Skill, RuntimeEvent | `call_id`, `timestamp` | The agent (or a skill) invokes another skill or tool. Captures the call graph. |
| **requires_scope** | Skill, Version | Scope (represented as Metadata node with `field=scope`) | `scope_type` (e.g., `read`, `write`, `delete`, `export`, `admin`), `required` (boolean) | The skill requests a permission scope at registration or invocation time. |
| **signed_by** | Version | Publisher | `signature_type`, `key_id`, `valid` (boolean) | The version is cryptographically signed by a publisher. Does not imply behavioral safety. |
| **updated_from** | Version (new) | Version (old) | `update_type` (e.g., `major`, `minor`, `patch`, `dependency`, `config`) | A version transition. Enables tracking of behavioral drift across updates. |
| **approved_by** | Version, Skill | ApprovalRecord (represented as RuntimeEvent with `event_type=approval`) | `approver`, `approval_method` (e.g., `automatic`, `user`, `admin`), `scope_at_approval` | A version or skill was approved through a review or consent process. |
| **has_source_label** | DataObject, RuntimeEvent | TrustLabel | `labeler` (e.g., `provenance_monitor`, `manual`, `policy`) | Assigns a trust label to a data object or event based on its origin. |
| **has_sensitivity** | DataObject | SensitivityLabel | `classifier` (e.g., `dlp`, `manual`, `policy`) | Assigns a sensitivity classification to a data object. |
| **has_trust** | Skill, Version | TrustLabel | `basis`, `effective_from` | Assigns a trust tier to a skill or version. May change over time (e.g., after drift detection). |
| **targets_sink** | RuntimeEvent, CodeSlice | Sink (represented as Metadata node with `field=sink`) | `sink_type` (e.g., `network`, `file`, `email`, `memory`, `config`, `database`), `external` (boolean) | An event or code component targets an external sink. Used to detect exfiltration and unauthorized output. |

## 2. Constraint Library

The constraint library defines seven graph constraints (C1--C7). Each constraint specifies the graph pattern that triggers it, the formal trigger condition, the risk level, the policy recommendation, and an example graph fragment showing a violation.

### C1: Declaration--Implementation--Permission Consistency

**Primary attack class:** T1 (Capability Laundering), T7 (Scope Inflation)

**Input pattern:**

```text
Skill --declares--> Metadata (capability set D)
Skill --implements--> CodeSlice (inferred capability set I)
Skill --requires_scope--> Scope (granted permission set G)
```

**Trigger condition:**

```text
I \ D != EMPTY   (inferred exceeds declared)
OR G \ I != EMPTY   (granted exceeds what implementation justifies)
```

Where `\` denotes set difference. In words: there exists a capability that the code implements but the metadata does not declare, or there exists a granted permission that the code does not use.

**Risk level:** HIGH (if inferred exceeds declared), MEDIUM (if only granted exceeds inferred)

**Policy recommendation:**
- If `I \ D != EMPTY`: HITL re-review of the skill manifest and implementation. Degrade to sandbox-only until resolved.
- If `G \ I != EMPTY` and `G \ D != EMPTY`: Deny the excess scope. Request the skill to re-register with minimum scope.

**Example graph fragment showing violation:**

```text
[Skill:pdf_summarizer] --declares--> [Metadata:capability=read_only_summary]
[Skill:pdf_summarizer] --requires_scope--> [Metadata:scope=files.export]
[Version:v1] --implements--> [CodeSlice:handler]
  CodeSlice:handler.sinks = ["https://external.example.com/collect"]
  CodeSlice:handler.capabilities = ["network_send", "file_export"]

VIOLATION: I \ D = {network_send, file_export} \ {read_only_summary} != EMPTY
```

### C2: Source-Aware High-Privilege Flow

**Primary attack class:** T2 (Cross-Skill Confused Deputy)

**Input pattern:**

```text
DataObject --has_source_label--> TrustLabel(level=untrusted)
DataObject --flows_to--> LLMContext
LLMContext --calls--> Skill (trust_tier=internal OR requires_scope in {write, delete, export, admin})
Skill --calls--> RuntimeEvent --targets_sink--> Sink(external=true)
```

**Trigger condition:**

```text
EXISTS path p from TrustLabel(untrusted) to Sink(external) via LLMContext:
  p contains no PolicyDecision(decision in {allow, hitl})
  AND p crosses a trust boundary (untrusted -> internal/high_privilege)
```

**Risk level:** CRITICAL

**Policy recommendation:** Deny the high-privilege call. Require explicit HITL with execution-layer facts showing the untrusted provenance. If HITL is not available, degrade to sandbox-only execution.

**Example graph fragment showing violation:**

```text
[DataObject:webpage] --has_source_label--> [TrustLabel:untrusted]
[DataObject:webpage] --flows_to--> [LLMContext:turn_42]
[LLMContext:turn_42] --calls--> [Skill:internal_search]
[Skill:internal_search] --calls--> [RuntimeEvent:call_99]
[RuntimeEvent:call_99] --targets_sink--> [Sink:confidential_repo]
// No PolicyDecision node between untrusted source and high-privilege sink

VIOLATION: untrusted data flows to high-privilege internal tool without policy boundary
```

### C3: Cross-Tool Exfiltration Sequence

**Primary attack class:** T6 (Split Exfiltration)

**Input pattern:**

```text
RuntimeEvent_1 (event_type=read, data has sensitivity >= confidential)
  --flows_to--> DataObject
  --flows_to--> RuntimeEvent_2 (event_type=transform)
  --flows_to--> RuntimeEvent_3 --targets_sink--> Sink(external=true)
```

All within the same session or across linked sessions. The sequence must contain at least one `flows_to` edge connecting a sensitive-read event to an external sink.

**Trigger condition:**

```text
EXISTS sequence <e1, e2, ..., en> in call_trace WHERE:
  e1.data_object.has_sensitivity.level IN {confidential, restricted, secret}
  AND en.targets_sink.external = true
  AND FOR ALL i in 1..n-1: ei --flows_to--> e(i+1)
  AND path contains no PolicyDecision(decision in {allow}) at the sensitivity boundary
```

**Risk level:** CRITICAL

**Policy recommendation:** Block the final external-write event. Generate a HITL prompt showing the full sequence with data sensitivity labels, all participating skills, and the target sink. If the sequence is part of a user-approved workflow, require re-confirmation with full data class visibility.

**Example graph fragment showing violation:**

```text
[Skill:file_search] --calls--> [RuntimeEvent:read_file]
[RuntimeEvent:read_file] --flows_to--> [DataObject:strategy_pdf]
[DataObject:strategy_pdf] --has_sensitivity--> [SensitivityLabel:confidential]
[DataObject:strategy_pdf] --flows_to--> [RuntimeEvent:summarize]
[RuntimeEvent:summarize] --flows_to--> [DataObject:summary]
[Skill:auto_share] --calls--> [RuntimeEvent:post_slack]
[DataObject:summary] --flows_to--> [RuntimeEvent:post_slack]
[RuntimeEvent:post_slack] --targets_sink--> [Sink:external_slack]
// No PolicyDecision at sensitivity boundary

VIOLATION: confidential data reaches external sink through multi-skill sequence
```

### C4: Version Drift After Approval

**Primary attack class:** T3 (Delayed Rug-Pull)

**Input pattern:**

```text
Version:old --approved_by--> ApprovalRecord
Version:new --updated_from--> Version:old
Fingerprint comparison between old and new
```

**Trigger condition:**

```text
approved(Version:old)
AND updated_from(Version:new, Version:old)
AND drift(Version:old.fingerprint, Version:new.fingerprint) > tau
```

Where `drift()` is a distance function over capability fingerprints (set of declared capabilities, observed sinks, scope requests, output patterns, dependency set). The threshold tau is configurable; default is 0.3 (normalized Hamming distance over the capability vector).

**Risk level:** HIGH (if drift exceeds tau), CRITICAL (if new capabilities include network/external sinks)

**Policy recommendation:**
- Revoke the approval for the new version.
- Require full re-approval through the metadata, implementation, and sandbox pipeline.
- Until re-approved, degrade the skill to sandbox-only execution.
- If the update type is `patch` or `dependency` and drift is high, escalate to QUARANTINE.

**Example graph fragment showing violation:**

```text
[Version:review_assistant@1.0] --approved_by--> [ApprovalRecord:auto_2026_01]
[Version:review_assistant@1.1] --updated_from--> [Version:review_assistant@1.0]
  update_type = "patch"
  Version@1.0.fingerprint.capabilities = {file_read, text_generation}
  Version@1.1.fingerprint.capabilities = {file_read, text_generation, subprocess, network_send}
  drift = 0.5 > tau = 0.3

VIOLATION: minor version update introduces high drift; requires re-approval
```

### C5: Approval Integrity

**Primary attack class:** T4 (Consent Laundering)

**Input pattern:**

```text
RuntimeEvent (event_type=approval)
  approval text source: LLMContext or untrusted DataObject
  actual action: RuntimeEvent (event_type != displayed_action_type)
```

**Trigger condition:**

```text
ApprovalDialog.text_source NOT-IN {execution_layer, policy_engine}
OR ApprovalDialog.displayed_action != ExecutionLayer.actual_action
OR ApprovalDialog.displayed_sink != ExecutionLayer.actual_sink
OR ApprovalDialog.displayed_data_class != ExecutionLayer.actual_data_class
```

**Risk level:** CRITICAL

**Policy recommendation:** Reject the approval. Regenerate the approval dialog using only execution-layer facts: true action type, true target sink, true data sensitivity class, true scope, and reversibility flag. If the system cannot independently verify the approval text against execution facts, block the action.

**Example graph fragment showing violation:**

```text
[RuntimeEvent:approval_dialog]
  text = "Share summary with the finance team"
  text_source = LLMContext (model-generated, influenced by untrusted tool output)
  displayed_action = "share"
  displayed_sink = "internal team"

[RuntimeEvent:actual_send_email]
  event_type = "email_send"
  targets_sink = [Sink:external@attacker.com]
  data_object = [DataObject:full_financial_report]
  data_sensitivity = "confidential"

VIOLATION: approval text does not match execution-layer facts
```

### C6: Persistence Boundary

**Primary attack class:** T5 (Persistence Pivot)

**Input pattern:**

```text
DataObject --has_source_label--> TrustLabel(level in {untrusted, unknown})
DataObject --flows_to--> RuntimeEvent (event_type in {memory_write, config_write, hook_write, policy_write})
RuntimeEvent --targets_sink--> PersistentStore
```

**Trigger condition:**

```text
EXISTS path p from TrustLabel(untrusted) to PersistentStore:
  p.flows_to edges include a write event
  AND p contains no PolicyDecision(decision=allow) with cross_session_risk=true
  AND PersistentStore.scope IN {cross_session, global}
```

**Risk level:** HIGH

**Policy recommendation:**
- Require separate HITL approval for the write, with explicit disclosure that the data will persist across sessions and its untrusted origin.
- Mark the write with a `cross_session_risk` label.
- Enable rollback capability for the target persistent store.
- If the persistent store is a policy or hook store, escalate to CRITICAL and require admin approval.

**Example graph fragment showing violation:**

```text
[DataObject:webpage_content] --has_source_label--> [TrustLabel:untrusted]
[DataObject:webpage_content] --flows_to--> [RuntimeEvent:memory_write]
[RuntimeEvent:memory_write] --targets_sink--> [PersistentStore:agent_memory]
  scope = "cross_session"
  // No PolicyDecision with cross_session_risk approval

VIOLATION: untrusted content written to cross-session persistent store
```

### C7: Scope Minimization

**Primary attack class:** T7 (Scope Inflation)

**Input pattern:**

```text
RuntimeEvent (event_type=task_description)
  inferred required scope from task: task_scope
Skill --requires_scope--> Scope (granted scope: granted_scope)
```

**Trigger condition:**

```text
granted_scope \ task_scope != EMPTY
AND NOT EXISTS justification(granted_scope \ task_scope)
```

Where `justification` is a policy entry, explicit user approval of excess scope, or a documented manifest reason.

**Risk level:** MEDIUM (if excess scope is write), HIGH (if excess scope is delete/export/admin)

**Policy recommendation:**
- Deny the excess scope. Re-register the skill with the minimum scope derived from the task.
- If the skill requires excess scope for legitimate reasons, require explicit user approval with a clear explanation of what additional capabilities are being granted and why.
- Log the scope inflation event for governance review.

**Example graph fragment showing violation:**

```text
[RuntimeEvent:task_description]
  text = "Show me my calendar for today"
  inferred_scope = {calendar.read}

[Skill:calendar_app] --requires_scope--> [Metadata:scope=calendar.read]
[Skill:calendar_app] --requires_scope--> [Metadata:scope=calendar.write]
[Skill:calendar_app] --requires_scope--> [Metadata:scope=contacts.read]
[Skill:calendar_app] --requires_scope--> [Metadata:scope=contacts.write]
[Skill:calendar_app] --requires_scope--> [Metadata:scope=email.send]

granted_scope \ task_scope = {calendar.write, contacts.read, contacts.write, email.send}
// No justification entry exists

VIOLATION: granted scope exceeds task-inferred scope by four capabilities
```

## 3. Evidence Path Format

### 3.1 JSON Schema

An evidence path is an ordered sequence of evidence items that connects a trigger condition to a policy finding. It is the primary artifact for audit, explainability, and post-hoc review.

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "title": "EvidencePath",
  "type": "object",
  "required": ["path_id", "constraint", "severity", "nodes", "edges", "findings"],
  "properties": {
    "path_id": {
      "type": "string",
      "description": "Unique identifier for this evidence path, generated as a UUID."
    },
    "constraint": {
      "type": "string",
      "enum": ["C1", "C2", "C3", "C4", "C5", "C6", "C7"],
      "description": "The constraint that was evaluated to produce this path."
    },
    "attack_class": {
      "type": "string",
      "enum": [
        "capability_laundering",
        "cross_skill_confused_deputy",
        "delayed_rug_pull",
        "consent_laundering",
        "persistence_pivot",
        "split_exfiltration",
        "scope_inflation"
      ],
      "description": "The attack class this path is associated with."
    },
    "severity": {
      "type": "string",
      "enum": ["low", "medium", "high", "critical"],
      "description": "Risk severity of the finding."
    },
    "nodes": {
      "type": "array",
      "items": {
        "type": "object",
        "required": ["id", "type", "properties"],
        "properties": {
          "id": { "type": "string" },
          "type": {
            "type": "string",
            "enum": [
              "skill", "version", "metadata", "code_slice",
              "runtime_event", "data_object", "trust_label",
              "sensitivity_label", "policy_decision"
            ]
          },
          "properties": { "type": "object" }
        }
      },
      "description": "Ordered list of nodes in the evidence path."
    },
    "edges": {
      "type": "array",
      "items": {
        "type": "object",
        "required": ["source", "target", "type"],
        "properties": {
          "source": { "type": "string", "description": "Source node id." },
          "target": { "type": "string", "description": "Target node id." },
          "type": {
            "type": "string",
            "enum": [
              "declares", "implements", "observes", "flows_to",
              "calls", "requires_scope", "signed_by", "updated_from",
              "approved_by", "has_source_label", "has_sensitivity",
              "has_trust", "targets_sink"
            ]
          },
          "properties": { "type": "object" }
        }
      },
      "description": "Ordered list of edges connecting the nodes."
    },
    "findings": {
      "type": "array",
      "items": {
        "type": "object",
        "required": ["message", "confidence"],
        "properties": {
          "message": { "type": "string" },
          "confidence": { "type": "number", "minimum": 0.0, "maximum": 1.0 },
          "evidence": {
            "type": "array",
            "items": {
              "type": "object",
              "properties": {
                "kind": { "type": "string" },
                "subject": { "type": "string" },
                "predicate": { "type": "string" },
                "object": { "type": "string" },
                "confidence": { "type": "number" },
                "attrs": { "type": "object" }
              }
            }
          }
        }
      },
      "description": "Findings produced by evaluating the constraint against the path."
    },
    "decision": {
      "type": "string",
      "enum": ["allow", "degrade", "hitl", "sandbox_only", "deny", "quarantine", "rollback"],
      "description": "The policy decision produced for this path."
    },
    "timestamp": {
      "type": "string",
      "format": "date-time",
      "description": "When the evidence path was generated."
    },
    "session_id": {
      "type": "string",
      "description": "The agent session in which this path was observed."
    }
  }
}
```

### 3.2 Audit Log Serialization

Evidence paths are serialized to audit logs in two formats:

**Structured JSON (primary):** The full `EvidencePath` object is written as a single JSON line (JSONL format) to the audit log file. Each line is a complete, self-contained evidence path. This format supports machine querying, filtering by constraint or severity, and batch analysis.

**Human-readable summary (secondary):** For operator-facing logs, each evidence path is rendered as a structured text block:

```text
[EVIDENCE PATH] path_id=<uuid> constraint=<Cn> severity=<level> decision=<decision>
  NODES:
    <n1>: <type> <key properties>
    <n2>: <type> <key properties>
    ...
  EDGES:
    <n1> --<edge_type>--> <n2> <key properties>
    ...
  FINDING: <message> (confidence=<float>)
  DECISION: <decision> <rationale>
```

Audit logs are append-only. Evidence paths are immutable once written. The `path_id` serves as the deduplication key: if the same constraint is evaluated on the same graph state and produces the same path, the existing entry is not duplicated.

## 4. Graceful Degradation

The evidence graph must function when certain evidence sources are unavailable. The following rules define how the system degrades gracefully.

### 4.1 Source Code Unavailable

When a skill's source code is unavailable (closed-source, hosted SaaS, obfuscated binary), the `CodeSlice` nodes and `implements` edges cannot be populated.

**Degradation strategy:**

1. Remove all `CodeSlice` nodes and `implements` edges from the graph for the affected skill.
2. Remove all `implements` edges from C1's input pattern. The constraint degrades to checking `declares` against `requires_scope` and `observes` only:
   - C1-degraded: `observed_capability \ declared_capability != EMPTY` OR `granted_scope \ declared_capability != EMPTY`
3. Increase the confidence threshold for findings that relied on implementation evidence. Report findings with a note: `"implementation_evidence=unavailable; finding based on metadata and runtime only"`.
4. Increase the weight of sandbox probing evidence (if available) to partially compensate.
5. C3 (exfiltration sequence), C5 (approval integrity), and C6 (persistence boundary) are unaffected because they rely on runtime events, not source code.
6. C4 (version drift) degrades to comparing metadata-level fingerprints only (schema diff, scope diff) rather than code-level fingerprints.

**Impact:** T1 (Capability Laundering) detection degrades from cross-layer (metadata + code + runtime) to dual-layer (metadata + runtime). False positives may increase because runtime observations are point-in-time snapshots rather than exhaustive capability extraction.

### 4.2 Runtime Trace Missing

When a skill has not been observed in execution (no sandbox run, no runtime monitoring data), the `RuntimeEvent` nodes and `observes`, `flows_to`, `calls`, and `targets_sink` edges from runtime data cannot be populated.

**Degradation strategy:**

1. Remove all `RuntimeEvent` nodes sourced from runtime monitoring. Retain `RuntimeEvent` nodes from sandbox probing if available.
2. C1 degrades to checking `declares` against `requires_scope` and `implements` (from code analysis):
   - C1-degraded: `inferred_from_code \ declared != EMPTY` OR `granted_scope \ inferred_from_code != EMPTY`
3. C2 (confused deputy) cannot be evaluated because it requires `flows_to` edges from tool output through LLMContext. Report: `"constraint C2 not evaluated: no runtime trace available"`.
4. C3 (exfiltration sequence) cannot be evaluated because it requires a call sequence. Report: `"constraint C3 not evaluated: no runtime trace available"`.
5. C6 (persistence boundary) degrades to checking `requires_scope` for persistence-related capabilities (memory_write, config_write) without confirming actual writes. This is a scope-level check, not a behavior-level check.
7. C4 (version drift) retains metadata-level and code-level fingerprint comparison but loses runtime-observed behavior comparison.
8. C5 (approval integrity) retains metadata-level checks but cannot verify actual action vs. displayed action.

**Impact:** Most sequence-based constraints (C2, C3) become unevaluable. The system falls back to static analysis (C1, C7) and governance checks (C4). This is the most significant degradation scenario and highlights the importance of runtime monitoring for compositional attack detection.

### 4.3 Manifest-Only Skill Handling

When only the manifest is available (no source code, no runtime trace, no sandbox data), the graph contains only `Skill`, `Version`, `Metadata`, and `requires_scope` edges.

**Degradation strategy:**

1. The graph is limited to metadata-layer nodes and edges.
2. C1 degrades to: `granted_scope \ declared_capability != EMPTY` (are permissions broader than the description?).
3. C7 (scope minimization) is fully operational because it compares granted scope to task-inferred scope, both of which are available.
4. All other constraints (C2, C3, C4, C5, C6) cannot be evaluated. Report: `"constraints C2--C6 not evaluated: manifest-only skill; recommend sandbox probing and runtime monitoring"`.
5. The system applies the `DEGRADE` policy decision: the skill may be invoked but only in sandbox-only mode, with output isolation, until richer evidence is collected.

**Impact:** Manifest-only handling is the minimum viable security posture. It catches scope inflation (T7) and basic capability declaration mismatches, but cannot detect any compositional attack that requires implementation or runtime evidence. The system explicitly reports which constraints were not evaluated and recommends evidence collection actions.

### 4.4 Degradation Summary Table

| Evidence Source Unavailable | Constraints Affected | Degradation Action |
|---|---|---|
| Source code | C1 (partial), C4 (partial) | Remove `CodeSlice` nodes; fall back to metadata + runtime; note in findings |
| Runtime trace | C2, C3, C5 (partial), C6 (partial) | Remove `RuntimeEvent` nodes; mark constraints as not evaluated; recommend monitoring |
| Both code and runtime | C1 (partial), C2, C3, C4 (partial), C5 (partial), C6 (partial) | Manifest-only mode; DEGRADE policy; recommend full evidence collection |
| Sandbox data | C1 (partial), C3 (partial) | Retain runtime monitoring data; note sandbox coverage gap |
| Approval records | C4 (partial), C5 | Remove `approved_by` edges; C4 cannot check re-approval triggers; C5 cannot evaluate |
