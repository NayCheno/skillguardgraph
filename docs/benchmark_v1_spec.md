# Benchmark v1 Specification

## Goal

Benchmark v1 is designed to address the key limitation of benchmark v0: generator-detector coupling and signal overlap between evidence layers. v1 introduces:

1. **Label-blind separation** — detection input excludes all label metadata.
2. **Hard negatives with 8 classes of legitimate high-risk behavior** — these are benign samples that exhibit patterns similar to malicious skills.
3. **Layer-specific attack samples** — samples where only one evidence layer can detect the attack.
4. **Independence** — generated by a separate script with different vocabulary.

## Structure

| Subset | Count | Description |
|---|---|---|
| Benign clean | 350 | Standard benign skills |
| Benign hard negatives | 350 | 8 classes of legitimate high-risk behavior |
| Malicious (7 classes × 100) | 700 | Layer-diverse malicious skills |
| **Total** | **1,400** | |

## Hard negative classes

| Class | Description | Count |
|---|---|---|
| Legitimate write | Tools that write files (note-taking, backup, export) | 50 |
| Legitimate export | Tools that send data externally (email, API, sharing) | 50 |
| Legitimate network retrieval | Tools that fetch web content (RSS, scraping, search) | 50 |
| Legitimate persistence | Tools that maintain state (session memory, config) | 45 |
| Admin maintenance | Tools with elevated scopes for admin tasks | 40 |
| CI/CD automation | Tools that trigger builds, deploy, or manage pipelines | 40 |
| Email/calendar automation | Tools that read/write email and calendar | 35 |
| Repository management | Tools that manage repos (merge, branch, release) | 40 |

## Layer-specific attack design

Each malicious sample must be detectable by at least one layer but NOT all layers:

| Pattern | Detectable by | Not detectable by |
|---|---|---|
| Metadata-only capability laundering | Metadata + C1 | Runtime (no trace) |
| Obfuscated code with hidden sink | Static analysis | Metadata (clean manifest) |
| Conditional network access | Sandbox probing | Static (branch not taken) |
| Cross-tool exfiltration chain | Runtime provenance | Static/Metadata (per-tool benign) |
| Version drift after approval | Version tracking (C4) | Runtime (same code path) |
| Tainted approval dialog | Approval integrity (C5) | Metadata/Static |
| Persistence from untrusted source | Persistence boundary (C6) | Metadata (legitimate-looking store) |
| Scope inflation | Scope alignment (C7) | Runtime (task completes within scope) |

## Label-blind requirements

When running detection, the following fields must be REMOVED from the input:

```
case_id, label, attack_class, expected_evidence,
success_validator, benign_pair, lifecycle_stages,
template_id, generator_name, hard_negative_class
```

## Acceptance criteria

| Criterion | Threshold |
|---|---|
| A2.1: Generator independence | v1 script does not import v0 generator |
| A2.2: Label-blind audit | Blind vs. unblind decision change = 0 |
| A2.3: Hard-negative FPR | FPR ≤ 0.05 on hard negatives |
| A2.4: Real-adapted cases | ≥ 30% unsafe cases adapted from real artifact patterns |
| A2.5: Validator coverage | Every unsafe case has a validator |
| A2.6: Label error rate | Human spot-check error rate ≤ 3% |

## Generation

```bash
cd experiments
PYTHONPATH=src python scripts/build_benchmark_v1.py
PYTHONPATH=src python scripts/run_label_blind_audit.py
```

## Relationship to v0

- v0 remains as a frozen sanity check.
- v1 is the primary evaluation benchmark for the paper.
- Both use the same fusion engine, policy constraints, and evidence graph.
