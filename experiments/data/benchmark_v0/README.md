## Label specification

- Each sample has `label` (`benign` or `malicious`), `attack_class`, `expected_evidence`, `success_validator`, `benign_pair`, `template_id`, `case_id`.
- Labels are generated by the benchmark generator (`experiments/scripts/build_benchmark.py`).
- The same generator was used to create the detector evaluation data.

## Known limitations

1. **Generator-detector coupling.** The benchmark generator and the policy engine share a common vocabulary of suspicious patterns (e.g., `is_external_sink`, `flows_to`, `declares_readonly_with_write_scope`). This creates a controlled evaluation environment but limits external validity.
2. **Perfect provenance labels.** Every synthetic trace includes correctly formatted `sink_type`, `is_external`, and `source_label` fields. Real platforms may omit or inconsistently apply these labels.
3. **No static/sandbox-only attack variants.** Every malicious sample has at least one runtime or metadata signal that alone suffices for detection. This means static and sandbox ablations show zero F1 drop.
4. **Template family.** All samples are generated from the same template family. Held-out and mutation stress checks remain within this family.

## Usage

- **Sanity check:** Verify that the artifact achieves expected metrics on v0.
- **Label-leakage audit:** Confirm that removing label fields does not change detector decisions.
- **Regression testing:** Use v0 to detect metric regressions during development.

## Prohibited usage

- **Do not** cite v0 results as the primary evaluation in the paper.
- **Do not** use v0 F1=1.0 as evidence of real-world detection capability.
- **Do not** use v0 ablation results to claim that static/sandbox layers are unnecessary.
